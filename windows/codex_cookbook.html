Prompts that keep Codex scoped, honest, and reviewable.

Four-line task prompt
Goal: <what must change>
Context: <files, folders, errors, screenshots>
Constraints: <architecture, style, safety, do-not rules>
Done when: <tests, behavior, docs, review conditions>
Use this when the task matters.It reduces guessing and makes the result easier to review.
Plan before coding/plan Propose a step-by-step migration plan before editing files.Complex, ambiguous, or high-stakes tasks.Review the plan, then let Codex implement.
Ask for interview
Before writing code, interview me. Ask the minimum questions needed to turn this into a precise implementation plan. Challenge unsafe assumptions.
When you know the goal but not the spec.Turns vague intent into a stronger task contract.
Debug failing test
Goal: make the failing test pass.
Context: paste the failure output and mention the suspected files.
Constraints: change the smallest surface area; do not update snapshots unless behavior is intentionally changed.
Done when: the narrow test passes and you explain root cause.
Use for surgical debugging.Keeps Codex from broad refactors.
Patch with proof
Implement the fix. After editing, run the narrowest relevant checks first, then the broader suite if cheap. Report: files changed, tests run, remaining risk.
Any implementation task.Makes verification explicit.
Review my branch
Review the current working tree. Prioritize correctness, security, data loss, race conditions, and API compatibility. Ignore style unless it affects behavior.
Before commit or PR.Pairs well with `/review`.
Refactor safely
Refactor without behavior changes. Preserve public APIs unless you justify the change first. Add/adjust tests only where they reveal existing behavior.
Refactors and migrations.Protects external contracts.
Document repo rules
Create or update AGENTS.md with: repo map, setup commands, test/lint commands, conventions, do-not rules, and done-when checklist. Keep it short and operational.
New repo or repeated mistakes.Reusable guidance is compounding leverage.
Retrospective into rules
We made the same mistake twice. Write a short retrospective: cause, guardrail, and exact AGENTS.md rule to prevent recurrence. Then patch AGENTS.md.

AGENTS.md starter

# AGENTS.md

## Repo map
- `src/`: application code
- `tests/`: automated tests
- `docs/`: user/developer docs

## Commands
- Install: `<package manager install command>`
- Test: `<narrow test command>` then `<full test command>`
- Lint/typecheck: `<lint command>` / `<typecheck command>`

## Engineering agreements
- Prefer small, reviewable diffs.
- Preserve public APIs unless the task explicitly says otherwise.
- Ask before adding new production dependencies.
- Never commit secrets, keys, tokens, or generated credentials.

## Done when
- Relevant tests pass or the reason they cannot run is documented.
- Changed behavior is documented.
- Remaining risk is listed clearly.
